# State (Zustand Stores)

Detail reference for `src/stores/`. Load when touching store APIs, filter semantics, undo, selection, or cross-store helpers.

## Stores

| Abstraction | Location | Purpose |
|-------------|----------|---------|
| useCanvasStore | stores/canvas-store.ts | Single canvas: ensureDefault (create if needed), selectedCanvasId |
| useTodoStore | stores/todo-store.ts | Todo list, CRUD, filtering, bulk operations (bulkSetCompleted, bulkSetStatus, bulkSetScheduled, bulkSetDeadline, bulkSetProject, bulkRemove), applyMutations (batch placement writes), addAt (positioned insertion with sortOrder), duplicate (copy task with assignments), purgeExpiredCompleted (with backup snapshot). `todosVersion: number` monotonic counter — bumped on every mutation that changes the id-set (add / addAt / duplicate / remove / bulkRemove / _restore / _removeNoUndo / purgeExpiredCompleted / load*); **not** bumped on field edits (update / toggleComplete / reorder / bulkSet*). Surfaces that only care about membership changes use `${todos.length}:${todosVersion}` as an O(1) cache key in place of the older `todos.map(t => t.id).sort().join(',')` — consumed by `ListView` + `HorizonsSlotContent` to gate the assignment-join-load effect |

| useFloatingTaskboardStore | stores/floating-taskboard-store.ts | Per-canvas placement widgets that render the singleton `Taskboard`. `loadByCanvas`, `add(canvasId, x, y)`, `updatePosition`, `updateSize`, `setCollapsed`, `remove` (with undo). | useTodoEventStore | stores/todo-event-store.ts | Lazy event-log cache feeding the discipline scoreboard + snooze graveyard widgets (stats-widgets-2026-04-25 Phase 4). Zustand store with `events: TodoEvent[]` + `loading` + `error` + three actions: `loadInRange(from, to, types?)` (windowed read used by the scoreboard's 12-week pull), `loadAll()` (full read used by the snooze graveyard), `invalidate()` (clears the cache). The two readers race in v1 (last caller wins on the shared `events` array — both subscribers re-pull on `useTodoStore.todos` reference change so steady state converges within one mutation; documented in P4 / P5 handoffs and acceptable for v1 per plan). Reads route through `todoEventRepository`; no write surface — events are produced by `todo-repository` inside the same transaction as the underlying todo write |
| useCanvasRailsStore | stores/canvas-rails-store.ts | Canvas side-rail layout: `rails: RailsState` (per-side `Rail | null` + widths/heights bags + optional `corners: Partial<Record<Corner, 'h'|'v'>>` — `'h'` lets the top/bottom rail claim the NW/NE/SW/SE corner cell; default/absent value `'v'` keeps the legacy "verticals own all four corners" layout; dangling claims are preserved in state and resolved at render time by `resolveCorner` — + optional `collapsed: Partial<Record<RailSide, boolean>>` — when `true`, `railSize` returns `COLLAPSED_RAIL_PX` (28 px) and `RailContainer` renders a slim icon-stub strip instead of slots; stored widths/heights survive so expand restores the prior size), `hydrated`, `pendingFocusSlotId` (transient focus target after split). Slot ops: `addRail`, `closeSlot`, `updateSlot` (patches the **active** tab's `listDefinitionId` and/or slot-level fields), `setSlotKind` (mutates the active tab's type with kind-appropriate seed via `SLOT_KINDS`; alias for `changeTabType` on the active tab), `setSlotOrientation(slotId, 'vertical'\|'horizontal')` + `setSlotWeekOffset(slotId, n)` (calendar slots only; wrappers over `updateSlot`), `dropSlotToSide` / `splitDropSlot` (delegate to pure `utils/rail-dnd` reducers), `splitSlot(slotId, dir)`, `createAndDockSlot(kind, listDefinitionId?)` (docks into first empty rail by priority right→left→top→bottom), `setRailSize`, `setSlotFlexBatch` (atomic per-slot flex writes used by `SlotDivider`), `setCornerOwner(corner, 'h'|'v')` / `clearCornerOwner(corner)`, `setRailCollapsed(side, boolean)` / `toggleRailCollapsed(side)` (collapsed bag entries are stored only when `true`; setting `false` removes the key and drops the whole bag when empty). Tab ops (rail-tabs): `addTab(slotId, kind, seed?)` (appends + activates), `closeTab(slotId, tabId)` (activates left sibling else right; cascade-closes the slot when `tabs` empties), `activateTab(slotId, tabId)`, `changeTabType(slotId, tabId, nextKind, seed?)`, `reorderTab(slotId, tabId, insertIdx)`, `moveTabToSlot(srcSlotId, tabId, destSlotId, insertIdx)`, `detachTabToNewSlot(srcSlotId, tabId, target)`. Float-dock ops (reverse of `popTabToCanvas` — see `utils/rail-dnd.ts` for the pure reducers): `dockFloatIntoSlot(descriptor, slotId, target: 'center' \| SplitZone, insertIndex?)` merges a float into an existing slot (`'center'` appends a tab at `insertIndex ?? tabs.length`; edge zones split off a new adjacent slot), and `dockFloatAsNewSlot(descriptor, { kind: 'empty-side', side } \| { kind: 'slot-split', slotId, zone })` docks a float onto an empty rail side or splits it off adjacent to an existing slot. Both reducers delete the source float row from its floating store (`useFloatingNoteStore` / `useFloatingCalendarStore` / `useFloatingTaskboardStore` / `useListInsetStore`) via fire-and-forget `removeFloatRow(descriptor)` after a successful rails update (detected by `applyDock* !== state.rails`) — a removal failure leaves a duplicate float visible but doesn't break the docked tab. Factory helpers `createLensSlot(listDefinitionId?)` / `createSlot(kind, listDefinitionId?)` / `createTaskboardSlot()` produce one-tab slots. `popSlotToCanvas` / `popTabToCanvas` extract the active/target tab into the matching floating store; both delegate to the exported `popTabAtPosition(tab, canvasId, x, y, init?)` helper (in `components/canvas/rails/RailsFrame.tsx`) so the menu pop-out path (upper-left-of-viewport + jitter via `computePopOutFlowPosition`) and the Phase 5 float-dock drag pop-out path (`pointerToFlowPosition` from `utils/rail-dnd.ts`) share one implementation — the menu path invokes it with `computePopOutFlowPosition()` coords, the drag path (`useRailsDragMonitor`'s tab-drag `zone.kind === 'canvas'` branch) invokes it with pointer-derived coords. Tab `seed` / `SlotPatch` carry `listDefinitionId` and (lens only) `runtimeFilterValue: number[]` — `updateSlot` patches the active tab's picks so a rail lens tab with `ListDefinition.runtimeFilter` keeps the user's choices across re-mount. `setTabRuntimeFilterValue(slotId, tabId, value: number[] \| undefined)` (lists-consistency P5 widened the value from a scalar) writes the picks; passing `undefined` *or* an empty array clears them. Taskboard tabs need no seed because the board is a singleton. Persisted through `settings.canvasRails` via `serializeRailsState` / `parseRailsState` (legacy `{ kind, listDefinitionId?, taskboardId? }` slots hydrate as single-tab slots; any persisted `taskboardId` on a tab is silently stripped; `collapsed` entries round-trip, non-`true` values + unknown sides are dropped) |

On `/list` the `runtimeFilterSpec` *is* round-tripped (hydrated by `applyDefinition`, flushed by `writeCurrentStateToDef` / `handleCreateNew`) so the user can also edit the spec from the toolbar's `Prompt` control; widget surfaces continue to edit the spec only via `DashboardListsEditor`. ## Filter Semantics

| Abstraction | Location | Purpose |
|-------------|----------|---------|
| useFilterStore | stores/filter-store.ts | Filter criteria (personIds, orgIds, projectIds, statusIds, tags as null\|Set of number ids; showCompleted, showHiddenStatuses (overrides hideByDefault), searchText, dateField, dateRangeStart/End, dateRangeIncludeNoDate, hasScheduled, hasDeadline; personFilterMode (include-orgs/direct-only), orgFilterMode (include-people/direct-only)); `setTags(Set<number>\|null)` toggles the tag-id clause (OR semantics via `assignedTagsMap`); displayed in TopBar filter bar |
| matchesFilter / applyFilter | stores/filter-store.ts | Top-level evaluators (not store methods). Take `filters: FilterCriteria` explicitly so the dashboard interpreter can synthesize a runtime FilterCriteria from a stored `TodoPredicate`. `statuses[]` drives hideByDefault exclusion; optional `today` anchors fuzzy-scheduled resolution. Tag filter evaluates against the per-todo `assignedTagIds` passed through from `applyFilter` (sourced from `useTagStore().assignedTagsMap`) — non-null + non-empty clause excludes tasks with no assignments |
| criteriaToPredicate / predicateToCriteria | stores/filter-store.ts | Converters between runtime `FilterCriteria` (Sets + Dates) and serializable `TodoPredicate` (arrays + ISO strings). Crossed at storage/evaluation boundaries (save-view serialize, dashboard-lists custom-membership evaluate) |
| OrgFilterMode | stores/filter-store.ts | Type: include-people, direct-only — include-people matches person-org + direct-org; direct-only matches only direct org assignment |
| PersonFilterMode | stores/filter-store.ts | Type: include-orgs, direct-only — include-orgs also matches tasks with directly-assigned orgs the filter person belongs to |
| computeFilterPersonOrgIds | stores/filter-store.ts | Helper: precomputes the set of orgs that filter persons belong to, for include-orgs person-filter matching (returns undefined when not applicable) |
