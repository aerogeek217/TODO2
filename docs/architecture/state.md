# State (Zustand Stores)

Detail reference for `src/stores/`. Load when touching store APIs, filter semantics, undo, selection, or cross-store helpers.

## Stores

| Abstraction | Location | Purpose |
|-------------|----------|---------|
| useCanvasStore | stores/canvas-store.ts | Single canvas: ensureDefault (create if needed), selectedCanvasId |
| useTodoStore | stores/todo-store.ts | Todo list, CRUD, filtering, bulk operations (bulkSetCompleted, bulkSetStatus, bulkSetScheduled, bulkSetDeadline, bulkSetProject, bulkRemove), applyMutations (batch placement writes), addAt (positioned insertion with sortOrder), duplicate (copy task with assignments), purgeExpiredCompleted (with backup snapshot). `todosVersion: number` monotonic counter — bumped on every mutation that changes the id-set (add / addAt / duplicate / remove / bulkRemove / _restore / _removeNoUndo / purgeExpiredCompleted / load*); **not** bumped on field edits (update / toggleComplete / reorder / bulkSet*). Surfaces that only care about membership changes use `${todos.length}:${todosVersion}` as an O(1) cache key in place of the older `todos.map(t => t.id).sort().join(',')` — consumed by `ListView` + `HorizonsSlotContent` to gate the assignment-join-load effect |
| useProjectStore | stores/project-store.ts | Projects for current canvas. `updateProjectGrouping(projectId, groupBy, groupOrder?)` is the single-field grouping setter consumed by `ProjectNode`'s ⊟ picker — wraps `projectRepository.updateGrouping` in `optimistic` (in-memory write + repository round-trip + snapshot revert on failure) |
| usePersonStore | stores/person-store.ts | People list, CRUD, todo-person assignments, bulk assign/unassign |
| useOrgStore | stores/org-store.ts | Orgs list, CRUD, assignedOrgsMap, personOrgMap (centralized person↔org membership), todo-org assignments (assign/unassign/bulk with undo) |
| useStatusStore | stores/status-store.ts | Status definitions: load, add, update, remove (cascade delete clears todos + default setting, with undo), reorder (drag sort with persisted sortOrder) |
| useTagStore | stores/tag-store.ts | Normalized tag registry (mirrors `useOrgStore` shape): `tags: Tag[]`, `assignedTagsMap: Map<number, Tag[]>`, `load`, `loadAssignments(todoIds)`, `add(name, color?)` (defaults to `DEFAULT_ENTITY_COLOR`), `update`, `remove` (with undo — re-hydrates cascaded `todoTags` joins), `assignTag`/`unassignTag`/`bulkAssignTag`/`bulkUnassignTag`, `getAssignedTags(todoId)`. Uniqueness is enforced case-insensitively at the repository layer; `#Foo` / `#foo` resolve to the same registry row |
| useListInsetStore | stores/list-inset-store.ts | List inset widgets CRUD; `add(listDefinitionId, canvasId, x, y)` pins an existing `ListDefinition` onto the canvas. The `ListInset` row carries an optional `runtimeFilterValue?: number[]` — the user's current OR-combined picks for a def with `ListDefinition.runtimeFilter` (lists-consistency P5 widened this from a scalar; the v41 Dexie migration lifts pre-P5 scalar values to single-entry arrays). `ListInsetNode`'s picker handler calls `update(inset)` with the key added / stripped (setting the value to `undefined` *or* an empty array deletes the key so the inset round-trips cleanly); switching the inset's `listDefinitionId` always strips the picks because the new def's runtime-filter field may be different or absent |
| useNoteStore | stores/note-store.ts | The single global "outside-tasks" note: `content`, debounced `setContent` + `flush`, `load()` seeds the row if missing. Canvas-pinned notes are placement-only and live in `useFloatingNoteStore` |
| useFloatingNoteStore | stores/floating-note-store.ts | Per-canvas placement widgets that render the single global note. `loadByCanvas`, `add(canvasId, x, y)`, `updatePosition`, `updateSize`, `remove` (with undo). Content stays in `useNoteStore`; this store only tracks x/y/w/h |
| useFloatingCalendarStore | stores/floating-calendar-store.ts | Per-canvas placement widgets that render the shared `CalendarStrip`. `loadByCanvas`, `add(canvasId, x, y)`, `updatePosition`, `updateSize`, `updateOrientation(id, 'vertical'\|'horizontal')`, `updateWeekOffset(id, n)` (clamped to ±`WEEK_OFFSET_MAX`), `remove` (with undo). Placement-only — parallels `useFloatingNoteStore`; `orientation`/`weekOffset` default to `'vertical'` / `0` at read time |
| useFloatingTaskboardStore | stores/floating-taskboard-store.ts | Per-canvas placement widgets that render the singleton `Taskboard`. `loadByCanvas`, `add(canvasId, x, y)`, `updatePosition`, `updateSize`, `setCollapsed`, `remove` (with undo). Entries live on the singleton `Taskboard` row; this store only tracks placement + collapse |
| useFloatingHorizonsStore | stores/floating-horizons-store.ts | Per-canvas placement widgets that render the horizon ribbon + selected-horizon list. `loadByCanvas`, `add(canvasId, x, y)`, `updatePosition`, `updateSize`, `setCollapsed`, `remove` (with undo). Ribbon state (`horizonSlots`, `selectedHorizon`) lives in settings; this store only tracks placement + collapse. Introduced in features-batch-2026-04 Phase 5 to back the `horizons` canvas widget kind that subsumes the retired Dashboard view |
| useCanvasRailsStore | stores/canvas-rails-store.ts | Canvas side-rail layout: `rails: RailsState` (per-side `Rail | null` + widths/heights bags + optional `corners: Partial<Record<Corner, 'h'|'v'>>` — `'h'` lets the top/bottom rail claim the NW/NE/SW/SE corner cell; default/absent value `'v'` keeps the legacy "verticals own all four corners" layout; dangling claims are preserved in state and resolved at render time by `resolveCorner` — + optional `collapsed: Partial<Record<RailSide, boolean>>` — when `true`, `railSize` returns `COLLAPSED_RAIL_PX` (28 px) and `RailContainer` renders a slim icon-stub strip instead of slots; stored widths/heights survive so expand restores the prior size), `hydrated`, `pendingFocusSlotId` (transient focus target after split). Slot ops: `addRail`, `closeSlot`, `updateSlot` (patches the **active** tab's `listDefinitionId` and/or slot-level fields), `setSlotKind` (mutates the active tab's type with kind-appropriate seed via `SLOT_KINDS`; alias for `changeTabType` on the active tab), `setSlotOrientation(slotId, 'vertical'\|'horizontal')` + `setSlotWeekOffset(slotId, n)` (calendar slots only; wrappers over `updateSlot`), `dropSlotToSide` / `splitDropSlot` (delegate to pure `utils/rail-dnd` reducers), `splitSlot(slotId, dir)`, `createAndDockSlot(kind, listDefinitionId?)` (docks into first empty rail by priority right→left→top→bottom), `setRailSize`, `setSlotFlexBatch` (atomic per-slot flex writes used by `SlotDivider`), `setCornerOwner(corner, 'h'|'v')` / `clearCornerOwner(corner)`, `setRailCollapsed(side, boolean)` / `toggleRailCollapsed(side)` (collapsed bag entries are stored only when `true`; setting `false` removes the key and drops the whole bag when empty). Tab ops (rail-tabs): `addTab(slotId, kind, seed?)` (appends + activates), `closeTab(slotId, tabId)` (activates left sibling else right; cascade-closes the slot when `tabs` empties), `activateTab(slotId, tabId)`, `changeTabType(slotId, tabId, nextKind, seed?)`, `reorderTab(slotId, tabId, insertIdx)`, `moveTabToSlot(srcSlotId, tabId, destSlotId, insertIdx)`, `detachTabToNewSlot(srcSlotId, tabId, target)`. Float-dock ops (reverse of `popTabToCanvas` — see `utils/rail-dnd.ts` for the pure reducers): `dockFloatIntoSlot(descriptor, slotId, target: 'center' \| SplitZone, insertIndex?)` merges a float into an existing slot (`'center'` appends a tab at `insertIndex ?? tabs.length`; edge zones split off a new adjacent slot), and `dockFloatAsNewSlot(descriptor, { kind: 'empty-side', side } \| { kind: 'slot-split', slotId, zone })` docks a float onto an empty rail side or splits it off adjacent to an existing slot. Both reducers delete the source float row from its floating store (`useFloatingNoteStore` / `useFloatingCalendarStore` / `useFloatingTaskboardStore` / `useListInsetStore`) via fire-and-forget `removeFloatRow(descriptor)` after a successful rails update (detected by `applyDock* !== state.rails`) — a removal failure leaves a duplicate float visible but doesn't break the docked tab. Factory helpers `createLensSlot(listDefinitionId?)` / `createSlot(kind, listDefinitionId?)` / `createTaskboardSlot()` produce one-tab slots. `popSlotToCanvas` / `popTabToCanvas` extract the active/target tab into the matching floating store; both delegate to the exported `popTabAtPosition(tab, canvasId, x, y, init?)` helper (in `components/canvas/rails/RailsFrame.tsx`) so the menu pop-out path (upper-left-of-viewport + jitter via `computePopOutFlowPosition`) and the Phase 5 float-dock drag pop-out path (`pointerToFlowPosition` from `utils/rail-dnd.ts`) share one implementation — the menu path invokes it with `computePopOutFlowPosition()` coords, the drag path (`useRailsDragMonitor`'s tab-drag `zone.kind === 'canvas'` branch) invokes it with pointer-derived coords. Tab `seed` / `SlotPatch` carry `listDefinitionId` and (lens only) `runtimeFilterValue: number[]` — `updateSlot` patches the active tab's picks so a rail lens tab with `ListDefinition.runtimeFilter` keeps the user's choices across re-mount. `setTabRuntimeFilterValue(slotId, tabId, value: number[] \| undefined)` (lists-consistency P5 widened the value from a scalar) writes the picks; passing `undefined` *or* an empty array clears them. Taskboard tabs need no seed because the board is a singleton. Persisted through `settings.canvasRails` via `serializeRailsState` / `parseRailsState` (legacy `{ kind, listDefinitionId?, taskboardId? }` slots hydrate as single-tab slots; any persisted `taskboardId` on a tab is silently stripped; `collapsed` entries round-trip, non-`true` values + unknown sides are dropped) |
| useTaskboardStore | stores/taskboard-store.ts | **Singleton** `board: Taskboard \| null`. `load()` populates `board` from the repository (no-ops when the row is absent); `ensureLoaded()` returns the row, seeding an empty one on first call. Entry ops: `add(todoId)`, `addAt(todoId, atIndex)` (positional insert with 1000-step `sortOrder` renormalization), `addMultipleAt(todoIds, atIndex)` (batch with even distribution), `removeEntry(todoId)`, `clear()`, `has(todoId)`, `reorder(fromIndex, toIndex)` (with undo). Entries persist inline on the singleton `Taskboard` row via `writeEntries`. No `taskboardId` / `defaultBoardId` — the model collapsed in the v33 migration |
| useListDefinitionStore | stores/list-definition-store.ts | Saved list definitions (the singular store after v39 folded `SavedView` in): load, add (appends with auto sortOrder; accepts `favorited?` / `maxTasks?` / `limitMode?` / `runtimeFilter?`), update, rename, setPinned, `setFavorited(id, next)` (writes the Favorites chip-bar flag independently of `pinnedToDashboard`), remove (with undo + snackbar), clone (unique-name copy), reorder. Exports `emptyPredicate()` for new custom lists. Backs every surface that used to split `ListDefinition` + `SavedView`: Dashboard cards (`pinnedToDashboard`), ListView Favorites chip bar (`favorited`), ListView Save / Load dialogs, rail-docked lens tabs, floating list insets. `ListDefinition.runtimeFilter?: RuntimeFilterSpec` declares a per-render picker field (`person`/`org`/`project`/`status`/`tag` — lists-consistency P5 added `tag`); the *value* is *not* stored on the def — each surface holds its own pick: `ListInset.runtimeFilterValue: number[]` (canvas float), `Tab.runtimeFilterValue: number[]` (rail lens tab), and — post-list-bugs P3 + lists-consistency P5 — ListView's local `runtimeFilterValue: number[] \| undefined` React state (transient, reset on every Load and never written back). All three are OR-combined arrays; an empty array is treated as no narrowing (helper passes the predicate through unchanged). On `/list` the `runtimeFilterSpec` *is* round-tripped (hydrated by `applyDefinition`, flushed by `writeCurrentStateToDef` / `handleCreateNew`) so the user can also edit the spec from the toolbar's `Prompt` control; widget surfaces continue to edit the spec only via `DashboardListsEditor`. Widget surfaces feed their picks into `buildDashboardLists` via `ctx.runtimeFilterValues: ReadonlyMap<defId, number[]>`; ListView bypasses the interpreter and calls `applyRuntimeFilter` directly against the `filter-store` criteria. Legacy saved-view translation helpers live in `data/saved-view-legacy.ts` (not re-exported here) — used only by v39 migration + restore, not by runtime |
| useUIStore | stores/ui-store.ts | Active view, selected task(s), multi-selection (selectedTodoIds Set, selectionAnchorId, selectionFocusId, focusedTodoId, rangeSelectTodo, selectAll), hoveredTodoId + setHoveredTodoId (cross-surface hover sync — TaskRow + MobileTaskRow read via selector, write on mouseenter/leave, expose `data-hovered-synced`), edit popup mode, bulk confirmation dialog state, inlineCreateAfterId (Insert hotkey), clipboard (clipboardTodoIds, clipboardSourceProjectId, cutTasks, clearClipboard), filteredListPopup (AttributeFilter: person/org), pendingCanvasTarget (command palette navigation), editingListDefId / editingListDefName + startEditingListDef / clearEditingListDef (drives ListView's "Editing preset" banner; cleared on Save / Cancel. Post-dashboard-lists-editor P2 there is no live caller — `DashboardListsEditor`'s "Edit in ListView…" deep-link was retired when the filter editor moved inline — the slice is kept as dead-stop until a future cleanup). **Lists editor open-state slice (lists-consistency P1):** `listsEditorOpen: boolean` + `listsEditorInitialId: number \| null` + `openListsEditor(initialId?)` + `closeListsEditor()`. Owned on the UI store so any component (canvas list widget's kind menu, rail lens slot's kind menu) can dispatch "open editor for def N" without prop-drilling; `CanvasPage` subscribes and renders `DashboardListsEditor` with `initialSelectedId={listsEditorInitialId ?? undefined}` when `listsEditorOpen === true`. `closeListsEditor` clears both fields atomically. The "Edit list" item on `WidgetKindMenu` (lens-only) routes through `useUIStore.getState().openListsEditor(defId)`. Float-drag slice: `floatDrag: { kind: FloatDragKind; id: number } \| null` + `setFloatDrag` (descriptor of the floating canvas widget currently being React-Flow-dragged, populated from `CanvasView.handleNodesChange` by decoding the React Flow node id prefix — `note-`/`calendar-`/`inset-`/`taskboard-`; setter is idempotent on identical kind+id so repeated `position` frames don't re-render downstream). `RailsFrame` reads `floatDrag !== null` to force-render `DockOverlay` even when no side is empty (docking into an occupied side creates a new slot, so the affordance stays useful). Companion `floatAnnouncement: string` + `setFloatAnnouncement` mirrors `useRailsDragMonitor`'s announcer pattern — `RailsFrame` renders it as a second `role=status aria-live=polite` region; values are `Dragging <kind>` on drag start, `Dropped in <zone>` on successful dock (set by `CanvasPage.handleFloatDock` via `describeFloatDockTarget`), cleared on non-dock release / cancel / stuck-state cleanup in `CanvasView`'s unmount effect |
| useUndoStore | stores/undo-store.ts | Undo/redo stacks (max 50), push/undo/redo/clear, isPerformingUndoRedo guard, beginGroup/endGroup (compound ops), snackbar state with auto-dismiss |
| useSettingsStore | stores/settings-store.ts | Theme mode (light/dark/system), theme color overrides (accent/canvasBg/surface/danger/warning/star/scheduled/deadline), defaultProjectId, defaultStatusId, quickStatusId (one-click status toggle, defaults to seeded Follow-up), seededAssignedStatusId, seededFollowupStatusId, completedRetentionDays, weekStartsOn (0=Sunday, 1=Monday — default 1; mirrored into `effective-date` via `setConfiguredWeekStart`), canvasViewport (single source of truth, debounced persistence); only user-customized colors set as inline overrides. Post-features-batch-2026-04 Phase 5: `dashboardUserLists` and `notesPinnedToDashboard` remain as dormant keys (the retired Dashboard view's persisted layout) for one release so rollback + older backups don't lose data; no reader in the app consumes them. `horizonSlots` + `selectedHorizon` continue to power the new `horizons` canvas widget kind |
| useFileStorageStore | stores/file-storage-store.ts | File storage connection state and actions; `pendingMigration` / `confirmMigration` / `cancelMigration` for legacy-import confirmation; exports refreshAllStores() |
| useFileOpsStore | stores/file-ops-store.ts | Data-ops store consumed by SettingsPage: `backups` list + `loadBackups` / `createBackup` / `deleteBackup` / `peekBackupData` / `restoreBackup` (delegates to `backupRepository`); `auditReport` + `runAudit` / `cleanupCurrentAudit` (delegates to `data/audit`); `restoreFromImport` (delegates to `data/restore`). Keeps the view free of direct `data/*` imports (Phase 2 layer-repair) |

## Filter Semantics

| Abstraction | Location | Purpose |
|-------------|----------|---------|
| useFilterStore | stores/filter-store.ts | Filter criteria (personIds, orgIds, projectIds, statusIds, tags as null\|Set of number ids; showCompleted, showHiddenStatuses (overrides hideByDefault), searchText, dateField, dateRangeStart/End, dateRangeIncludeNoDate, hasScheduled, hasDeadline; personFilterMode (include-orgs/direct-only), orgFilterMode (include-people/direct-only)); `setTags(Set<number>\|null)` toggles the tag-id clause (OR semantics via `assignedTagsMap`); displayed in TopBar filter bar |
| matchesFilter / applyFilter | stores/filter-store.ts | Top-level evaluators (not store methods). Take `filters: FilterCriteria` explicitly so the dashboard interpreter can synthesize a runtime FilterCriteria from a stored `TodoPredicate`. `statuses[]` drives hideByDefault exclusion; optional `today` anchors fuzzy-scheduled resolution. Tag filter evaluates against the per-todo `assignedTagIds` passed through from `applyFilter` (sourced from `useTagStore().assignedTagsMap`) — non-null + non-empty clause excludes tasks with no assignments |
| criteriaToPredicate / predicateToCriteria | stores/filter-store.ts | Converters between runtime `FilterCriteria` (Sets + Dates) and serializable `TodoPredicate` (arrays + ISO strings). Crossed at storage/evaluation boundaries (save-view serialize, dashboard-lists custom-membership evaluate) |
| OrgFilterMode | stores/filter-store.ts | Type: include-people, direct-only — include-people matches person-org + direct-org; direct-only matches only direct org assignment |
| PersonFilterMode | stores/filter-store.ts | Type: include-orgs, direct-only — include-orgs also matches tasks with directly-assigned orgs the filter person belongs to |
| computeFilterPersonOrgIds | stores/filter-store.ts | Helper: precomputes the set of orgs that filter persons belong to, for include-orgs person-filter matching (returns undefined when not applicable) |
