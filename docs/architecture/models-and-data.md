# Models & Data Layer

Detail reference for `src/models/` (TypeScript interfaces) and `src/data/` (Dexie persistence). Load when touching schema, repositories, migrations, import/export, or model shapes.

## Models

| Abstraction | Location | Purpose |
|-------------|----------|---------|
| Canvas | models/canvas.ts | Named spatial workspace |
| Project | models/project.ts | Positioned group of tasks on a canvas (optional color). Optional `groupBy?: ProjectGroupBy \| null` (`'status' \| 'people' \| 'org' \| 'tag' \| 'scheduled' \| 'deadline' \| 'date'`) drives `SortableTaskList`'s grouped render path — null/undefined → flat list. Optional `groupOrder?: string[]` reserves slot for explicit group ordering; v1 has no UI for it (alphabetical / sortOrder fallback). Both fields are stored inline, not indexed |
| ProjectGroupBy | models/project.ts | Union of the seven group-by dimensions accepted by `Project.groupBy` (lists-consistency P4 added `'tag'`). Validated in `import-validation.ts` via `VALID_PROJECT_GROUP_BY` so legacy / hand-edited backups can't smuggle in unknown keys |
| TodoItem | models/todo-item.ts | Core todo entry (id optional, pre-insert); optional `progress`, `statusId`, `scheduledDate` (ScheduledValue), `dueDate` (deadline), `recurrenceRule` |
| PersistedTodoItem | models/todo-item.ts | TodoItem with guaranteed id (post-insert) |
| ScheduledValue | models/scheduled-value.ts | Discriminated union for `scheduledDate`: `{kind:'date', value:Date}` or `{kind:'fuzzy', token: FuzzyToken}` where FuzzyToken ∈ today/tomorrow/this-week/next-week/this-month/next-month |
| RecurrenceRule | models/recurrence.ts | Recurrence pattern: type (daily/weekly/biweekly/monthly/quarterly/yearly), optional originalDayOfMonth to prevent drift |
| ListDefinition, RuntimeFilterSpec, RuntimeFilterField | models/list-definition.ts | Saved list definition: name, sortOrder, `pinnedToDashboard`, `favorited: boolean` (renders the def in ListView's Favorites chip bar; independent of Dashboard pin), membership (only `custom` post-v24 — carries a `TodoPredicate`), sort, grouping, optional `maxTasks?: number` + `limitMode?: 'hard' \| 'scroll'` (visible-task cap, persisted per def; drives ListView's hard-truncate vs scroll-region rendering). Optional `runtimeFilter?: { field: 'person'\|'org'\|'project'\|'status'\|'tag'; label? }` declares a per-render picker merged into the predicate at interpret time via `applyRuntimeFilter` in `services/dashboard-lists.ts` (lists-consistency P5 added `'tag'`; the user-facing label was renamed "Runtime filter" → "Prompt" in P2 — code identifiers unchanged). Not persisted on the def — each rendering surface stores its current OR-combined picks (`ListInset.runtimeFilterValue: number[]` / `Tab.runtimeFilterValue: number[]`) and passes the map to `buildDashboardLists`. v39 folded the retired `SavedView` model into this one — every pre-v39 saved view became a `favorited: true`, `pinnedToDashboard: false` list definition |
| TodoPredicate | models/filter-predicate.ts | Serializable predicate DSL mirroring live filter fields (number arrays + ISO strings — no Sets, no Dates). Stored inside `ListMembership.custom` (and legacy `LegacySavedViewFilters` for pre-v39 imports); runtime converts to `FilterCriteria` via `predicateToCriteria` at evaluation time |
| Person | models/person.ts | Assignable person with name + initials. Display color is derived at render time from the person's first assigned org via `utils/person-color.resolvePersonColor` (the legacy `color` field was dropped in Dexie v31) |
| PersistedPerson | models/person.ts | Person with guaranteed id (post-insert) |
| Org | models/org.ts | Organization/group for people (name, optional initials, optional color) |
| PersonOrg | models/person-org.ts | Many-to-many join: person ↔ org |
| TodoPerson | models/todo-person.ts | Many-to-many join: todo ↔ person |
| TodoOrg | models/todo-org.ts | Many-to-many join: todo ↔ org (direct org assignment) |
| Status | models/status.ts | User-defined workflow state: name, color, sortOrder, `icon` (key from StatusIcon registry, default 'circle'), optional `hideByDefault` (excluded from default filter when true) |
| PersistedStatus | models/status.ts | Status with guaranteed id (post-insert) |
| Tag | models/tag.ts | First-class normalized tag: `{id?: number, name: string, color: string}`. Canonical casing is whatever the creator entered; uniqueness is enforced case-insensitively at the repository level. Assigned via `todoTags` join, never inlined on `TodoItem` |
| PersistedTag | models/tag.ts | Tag with guaranteed id (post-insert) |
| TodoTag | models/todo-tag.ts | Many-to-many join: todo ↔ tag |
| ListInset | models/list-inset.ts | Canvas widget referencing a `ListDefinition` (`listDefinitionId` FK); delegates to the dashboard-lists interpreter. Optional `runtimeFilterValue?: number[]` holds the user's current OR-combined picks for a def with `ListDefinition.runtimeFilter` so re-opening the canvas keeps the picks (lists-consistency P5 widened from a scalar; the v41 Dexie migration lifts pre-P5 scalar values to single-entry arrays). Ignored when the referenced def has no runtime filter; `ListInsetNode` strips the key on list-def switch and on empty-array writes |
| Note | models/note.ts | The single global markdown note: `{content, createdAt, modifiedAt}`. One row backs the dashboard tile, the rail Notes slot, and every canvas `FloatingNote` placement |
| FloatingNote | models/floating-note.ts | Canvas-pinned placement that renders the single global note. `{canvasId, x, y, width, height}`. Content lives in `Note`; this row only tracks placement. Dexie secondary index on `canvasId` |
| FloatingCalendar | models/floating-calendar.ts | Canvas-pinned placement that renders the shared two-week calendar strip. `{canvasId, x, y, width, height}`. Dexie secondary index on `canvasId` |
| FloatingTaskboard | models/floating-taskboard.ts | Canvas-pinned placement that renders the singleton `Taskboard`. `{canvasId, x, y, width, height, collapsed?}`. Dexie secondary index on `canvasId`. Pre-v33 rows carried a `taskboardId`; the v33 migration strips it from disk and persisted exports |
| FloatingHorizons | models/floating-horizons.ts | Canvas-pinned placement that renders the horizon ribbon + selected-horizon list (retired Dashboard view's content). `{canvasId, x, y, width, height, collapsed?}`. Ribbon state lives in `settings.horizonSlots: number[]` (post-stats-widgets-2026-04-25 P6 — array of `ListDefinition.id`s replacing the prior `Partial<Record<HorizonKey, number>>` map) + `settings.selectedHorizonDefId: number | null` — this row is placement-only. Dexie secondary index on `canvasId` (table added in v38) |
| FloatingStatus | models/floating-status.ts | Canvas-pinned placement that renders the open-by-status stacked-bar widget (stats-widgets-2026-04-25 P1). `{canvasId, x, y, width, height, collapsed?}`. Body content is derived from live `todo-store` + `status-store` via `selectStatusBreakdown` — placement-only, no per-widget settings. Dexie secondary index on `canvasId` (table added in v43) |
| FloatingScoreboard | models/floating-scoreboard.ts | Canvas-pinned placement that renders the discipline scoreboard (defer / completion / lag) widget (stats-widgets-2026-04-25 P1). `{canvasId, x, y, width, height, collapsed?}`. Body content is derived from `todo-events` + `todo-store` via `selectDisciplineMetrics` — placement-only. Dexie secondary index on `canvasId` (table added in v44) |
| FloatingSnoozeGraveyard | models/floating-snooze-graveyard.ts | Canvas-pinned placement that renders the snooze graveyard (top-N most-rescheduled open todos) widget (stats-widgets-2026-04-25 P1). `{canvasId, x, y, width, height, collapsed?}`. Body content is derived from `todo-events` + `todo-store` via `selectMostDeferred` — placement-only. Dexie secondary index on `canvasId` (table added in v45) |
| TodoEvent, PersistedTodoEvent, TodoEventType | models/todo-event.ts | Append-only history log entry (stats-widgets-2026-04-25 P3). One row per tracked-field mutation on a todo (`scheduledDate` / `dueDate` / `statusId` / `isCompleted`) plus a `created` event at insert time. `TodoEventType ∈ {'created' \| 'scheduled' \| 'deadline' \| 'status' \| 'completed' \| 'reopened'}`. `fromValue` / `toValue` encoding: ISO date string for fixed dates / `'fuzzy:<token>'` for `ScheduledValue.kind === 'fuzzy'` / numeric `statusId` for status / null for unset / null on both sides for `created` / `completed` / `reopened` (the type carries the meaning). Idempotent — events are emitted only when the value actually changes (`from !== to`). Backing store: `db.todoEvents` (Dexie v42, `++id, todoId, type, timestamp`) |
| Taskboard | models/taskboard.ts | **Singleton** queue of tasks: `{entries: TaskboardEntry[], createdAt, updatedAt}`. Entries live inline (no join table) — every entry-level mutation rewrites the row. The `taskboards` Dexie table always holds exactly one row (the v33 migration coalesces any pre-v33 multi-row table into a single row and strips the `name` field). Referenced by the dashboard card, every rail-slot taskboard tab, and every `FloatingTaskboard` widget, so the same board renders live across all surfaces |
| TaskboardEntry | models/taskboard-entry.ts | Ordered slot within the singleton `Taskboard`: `{todoId, sortOrder}`. Stored inline on the Taskboard row — `todoId` uniquely locates an entry |
| Slot, Tab, TabType, SlotKind, Rail, RailsState, Corner, CornerOwner | models/canvas-rails.ts | Canvas-side-rail shapes. `SlotKind ∈ {'lens','notes','calendar','taskboard','horizons'}` (canonical `SLOT_KINDS`); `TabType` is an alias used by per-tab content type. `Slot` is `{ id, flex?, tabs: Tab[], activeTabId, orientation?, weekOffset? }` — every slot carries ≥1 `Tab { id, type: TabType, listDefinitionId?, runtimeFilterValue?: number[] }` (lens tabs only — `runtimeFilterValue` is the OR-combined picks for a def with `ListDefinition.runtimeFilter`; post-v33; lists-consistency P5 widened from a scalar and the v41 migration lifts persisted scalars to single-entry arrays — `parseTab` accepts both shapes for forward-compat with pre-v41 imports; legacy `taskboardId` is silently stripped on parse). `getActiveTab(slot)` resolves the active tab (falls back to `tabs[0]` if `activeTabId` is stale). `RailsState` has per-side `Rail | null`, optional `widths`/`heights` bags (persisted even when the rail is null), and optional `corners: Partial<Record<Corner, CornerOwner>>` where `Corner ∈ {nw,ne,sw,se}` and `CornerOwner ∈ {'h','v'}` (`'h'` lets the top/bottom rail extend into the corner cell; default/absent `'v'` keeps the legacy vertical-owns layout). `resolveCorner(rails, corner)` falls back to the orthogonal side when the stored owner's rail is absent; `computeRailGridArea(rails, side)` derives each rail's grid-area in the `RailsFrame` 3×3 grid. `cornerForSideClaim(side, 'start'\|'end')` maps an empty-side drop-strip sub-zone to the corner the new rail would claim. Helpers: `clampRailSize` ([200, 600]), `defaultRailSize`, `railSize`, `railOrientationForSide`, `EMPTY_RAILS`, `CORNERS`, `serializeRailsState` / `parseRailsState` (accepts both the new `{ tabs, activeTabId }` shape **and** the legacy `{ kind, listDefinitionId?, taskboardId? }` slot shape — legacy is wrapped into a single-tab slot and `taskboardId` is dropped; round-trips `corners`, dropping unknown keys and non-`'h'|'v'` values; returns null on shape failure) |
| Backup | models/backup.ts | Auto-snapshot record: trigger type, serialized data, size |
| LegacySavedView, LegacySavedViewFilters | data/saved-view-legacy.ts | Retired `SavedView` shape kept as a data-layer-only type so the v39 migration + legacy-import restore can translate pre-v39 rows. Not exported from `models/`; runtime has no `useSavedViewStore` anymore |
| AppView | models/app-view.ts | Enum: Canvas, Dashboard, List, Calendar, Settings |
| ListSortBy | models/app-view.ts | Type: name, date, scheduled, deadline, people, org, project, status. Used by list-definition `sort.by`. `'name'` is `/list`-only (lists-consistency P3); dashboard widget surfaces fall back to sortOrder via `compareBySortBy`'s default branch |
| ListGroupBy | models/app-view.ts | Type: `'none'` + every `ListSortBy` value except `'name'` (which is sort-only — a tag-by-name grouping is meaningless). Drives `ListView` section-builder choice; `'none'` renders a flat list (header hidden). `applyDefinition` coerces an incoming `'name'` grouping (defensive — pre-P3 backups can't carry it, but a hand-edited def could) to `'none'` |
| ListItemSortBy | models/app-view.ts | Type: `'manual' \| 'name' \| 'date' \| 'scheduled' \| 'deadline'`. Sort applied within each group (or across the whole list when groupBy='none'). Round-trips into saved list-definitions via `encodeGroupSort` / `resolveGroupBy` / `resolveItemSortBy` (lists-consistency P3 widened the union with `'name'`; `resolveSavedViewGrouping` in `data/saved-view-legacy.ts` translates a legacy `sortBy: 'name'` into a `groupBy: 'date'` fallback because `'name'` is sort-only) |
| DateField | models/app-view.ts | Type: date, scheduled, deadline, created, modified — used by filter store and saved views. `'date'` filters on `effectiveDate`; `'scheduled'` on resolved `scheduledDate`; `'deadline'` on `dueDate` |
