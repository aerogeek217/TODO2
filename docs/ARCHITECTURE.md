# Architecture Overview

This file is the always-loaded orientation. For per-abstraction detail, load the relevant reference file listed under [Detail References](#detail-references).

## Tech Stack
- **Frontend**: React 19 + TypeScript, built with Vite
- **State**: Zustand stores
- **Database**: Dexie.js over IndexedDB (local-first)
- **Spatial Canvas**: React Flow
- **Drag-and-Drop**: dnd-kit
- **Routing**: React Router v7

## Data Flow

1. **App startup**: Dexie opens IndexedDB ‚Üí `useCanvasStore.ensureDefault()` ‚Üí `fileStorageService.initialize()` reconnects to saved file handle and loads file data into IndexedDB. Settings store loads theme mode and applies `data-theme` to `<html>`; only user-customized color overrides are set as inline styles. Expired completed tasks purged if `completedRetentionDays` configured. `backupScheduler.start()` begins periodic auto-snapshots.
2. **Normal use**: User acts in UI ‚Üí Zustand stores call repository methods ‚Üí Dexie persists to IndexedDB. If file storage is connected, Dexie hooks debounce-save all changes to the JSON file on disk. Cross-cutting task capture rides `QuickAddBar` (overlays/) ‚Äî `Ctrl+Space` / FAB / `command-registry.openQuickAdd` flips `ui-store.quickAddOpen` and the bar's parser-driven flow runs `useTodoStore.add` + `applyNlpMetadata` on submit; the older `TaskEditPopup` create modal stays reachable only via the bar's Details / Tab handoff and the canvas right-click "New Task" menu.
3. **Canvas**: Single canvas only; `useCanvasStore.ensureDefault()` creates/selects it at startup.
4. **View switching**: Sidebar icons navigate between Canvas, List, Calendar, Settings via React Router. The horizon ribbon + selected-horizon list (5 slots mapped via `settings.horizonSlots` to seeded custom-predicate `listDefinitions`: This week / Next week / Rest of month / Later / Someday) is a canvas widget (`horizons` kind) ‚Äî drag onto any rail or drop it as a floating node. List view groups all todos by a user-selected sort-by attribute.
5. **Person assignment**: Many-to-many via `todoPeople` join; `usePersonStore` maintains an `assignedPeopleMap` cache.
6. **Person-org membership**: Many-to-many via `personOrgs` join; a person can belong to multiple orgs.
